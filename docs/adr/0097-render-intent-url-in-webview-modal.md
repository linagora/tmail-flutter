# 97. Render Intent URL in WebView Modal

Date: 2026-05-29

## Status

Accepted

## Context

Step 4 of [ADR-0095](0095-external-drive-file-picker-integration.md). After obtaining the intent URL from `POST /intents`, TMail must render it in a sandboxed browser context so the drive app's file-picker UI loads and can communicate back via `postMessage`.

Platform split:
- **Web**: Flutter renders as a DOM `<iframe>` via `HtmlIframeWidget` — real cross-origin frame, standard `postMessage` applies.
- **Mobile/tablet**: `flutter_inappwebview`'s `InAppWebView` — native WKWebView/WebView, no cross-frame boundary; `window.parent === window`, so a shim is required (see [ADR-0096](0096-postmessage-handshake-for-drive-intent-webview.md)).

During development (before the real intent flow is wired), a `data:` URI generated by `DriveIntentFakePage` is used to exercise the full modal + postMessage path on both platforms without a live drive server.

Reference implementation: [PR #4448](https://github.com/linagora/tmail-flutter/pull/4448) (hackathon POC — not for direct merge; used to validate the approach).

## Decision

### 1. File Structure

`DriveIntentWebViewModal` is split across three files:

```
workplace/lib/presentation/view/
  drive_intent_web_view_modal.dart         # conditional export
  drive_intent_web_view_modal_mobile.dart  # InAppWebView variant
  drive_intent_web_view_modal_web.dart     # HtmlIframeWidget variant
  drive_intent_web_view_modal_shell.dart   # shared chrome (close button only)
  drive_intent_fake_page.dart              # dev/test data: URI page
```

The conditional export resolves the platform variant at compile time:

```dart
// drive_intent_web_view_modal.dart
export 'drive_intent_web_view_modal_mobile.dart'
    if (dart.library.html) 'drive_intent_web_view_modal_web.dart';
```

### 2. Widget Structure

Both platform variants are `StatefulWidget` — they mix in `DriveIntentMessageHandlerMixin` (and the web variant also mixes in `WebWindowMessageMixin`). Both delegate chrome to `DriveIntentWebViewModalShell`.

**Call site (all platforms):**
```dart
showDialog<List<DriveDocument>?>(
  context: context,
  builder: (_) => DriveIntentWebViewModal(
    url: intentUrl,
    intentId: intentId,
    onRegisterExternalHandler: externalHandlerRegistrar, // null on mobile
  ),
);
```

`DrivePickerStateMixin` owns the `showDialog` call and the `_modalOpen` guard that prevents double-opening.

**Shared constructor:**
```dart
class DriveIntentWebViewModal extends StatefulWidget {
  const DriveIntentWebViewModal({
    super.key,
    required this.url,
    required this.intentId,
    this.onRegisterExternalHandler, // ignored on mobile; used by web (ADR-96)
  });
  final Uri url;
  final String intentId;
  final OnRegisterExternalHandler? onRegisterExternalHandler;
}
```

`OnRegisterExternalHandler` and `DriveMessageHandler` typedefs live in `drive_intent_web_view_modal_shell.dart`.

### 3. Shell — `DriveIntentWebViewModalShell`

A shared `StatelessWidget` that wraps both platform variants with consistent chrome:

```
PointerInterceptor                    // required on web; no-op on mobile
  GestureDetector (onTap: close)      // backdrop tap dismisses
    Dialog (
      backgroundColor: white,
      shape: RoundedRectangleBorder(radius: 6),
      child: ConstrainedBox(maxWidth: 800, maxHeight: 677)
        GestureDetector (onTap: {})   // stops backdrop propagation inside dialog
          Column
            SizedBox(height: 14)
            close button (X icon, top-right)
            Expanded(child: <platform WebView>)
    )
```

The dialog is **not** full-screen — it is constrained to `maxWidth: 800 / maxHeight: 677` for a consistent desktop-like appearance on all platforms. `PointerInterceptor` wraps the entire dialog; it is a no-op on mobile so there is no behavioural difference.

There are **no footer buttons** in the shell. "Add as link" / "Add as attachment" labels are passed to the drive platform via the `actions` field in `POST /intents` (see ADR-0095 §3), so the drive platform renders those choices inside its own WebView UI.

### 4. Web — `HtmlIframeWidget`

Uses `HtmlIframeWidget` from `package:core` which wraps `HtmlElementView` internally. The `onIframeCreated` callback provides the `IFrameElement` reference needed for sending ack messages:

```dart
HtmlIframeWidget(
  key: ValueKey(widget.intentId),   // forces recreation per intent session
  src: widget.url.toString(),
  onIframeCreated: (iframe) {
    _iframeElement = iframe;
  },
),
```

`ValueKey(intentId)` ensures the element is recreated if a new intent session starts without closing the modal.

### 5. Mobile / Tablet — `InAppWebView`

Uses `InAppWebView` from `flutter_inappwebview` (already present in the project). Two URL modes:

- **Real URL:** `initialUrlRequest: URLRequest(url: WebUri.uri(widget.url))` 
- **`data:` URI (dev/test):** `initialData: InAppWebViewInitialData(data: DriveIntentFakePage.buildHtml(widget.intentId))` — detected via `widget.url.scheme == 'data'`

```dart
InAppWebView(
  key: ValueKey(widget.intentId),
  initialUrlRequest: _isDataUri ? null : URLRequest(url: WebUri.uri(widget.url)),
  initialData: _isDataUri
      ? InAppWebViewInitialData(data: DriveIntentFakePage.buildHtml(widget.intentId))
      : null,
  initialSettings: InAppWebViewSettings(),
  initialUserScripts: UnmodifiableListView([
    UserScript(
      source: DriveIntentShims.parentPostMessageShim,  // see ADR-0096 §2
      injectionTime: UserScriptInjectionTime.AT_DOCUMENT_START,
    ),
  ]),
  onWebViewCreated: (controller) {
    _webViewController = controller;
    controller.addJavaScriptHandler(
      handlerName: DriveIntentShims.handlerName,
      callback: (args) => onMessage(
        raw: args[0] as String,
        origin: args.length > 1 ? args[1] as String? : null,
      ),
    );
  },
);
```

`AT_DOCUMENT_START` injection is critical — injecting later (`AT_DOCUMENT_END`) risks a race where the drive app's `window.parent.postMessage` call fires before the shim is in place.

### 6. Dev / Test — `DriveIntentFakePage`

`DriveIntentFakePage` generates a self-contained HTML page as a `data:` URI. It simulates the full handshake: sends `intent-{id}:ready` on load, waits for ack, then sends `intent-{id}:done` with a single fake `DriveDocument` after 3 s. Uses `targetOrigin: '*'` because the parent's actual origin is not known from a data: context.

`DriveAttachmentUriValueNotifierProvider` returns a `DriveIntentFakePage` data: URI when the real platform URL is unavailable, keeping the button and modal exercisable during development.

### 7. Intent ID as Session Key

The `intentId` returned by `POST /intents` is unique per drive session. It is passed through `ValueKey` (forces widget recreation) and used as the `_intentId` in `DriveIntentMessageHandlerMixin` (guards postMessage callbacks against stale sessions). No separate session key is needed.

### 8. Lifecycle & Cleanup

On any terminal event (`done` / `error` / `cancel` / close button):

1. `closeModal(result)` is called (from `DriveIntentMessageHandlerMixin`).
2. `onCleanup()` runs — web variant removes the window listener via `stopWindowMessageListener()`.
3. `Navigator.of(context).pop(result)` dismisses the dialog.

The web modal overrides `dispose()` to call `onCleanup()` as a safety guard for cases where the modal is removed from the widget tree without going through `closeModal` (system back, parent navigation, etc.). `onCleanup` is idempotent so double-calls are safe.

### 9. Security Constraints

- Intent URL `https://` scheme is validated before passing to `InAppWebView` / `HtmlIframeWidget` (enforced in Step 3, ADR-0095 §3).
- The `window.parent` shim (mobile) only forwards messages to Flutter — it does not expose any Flutter internals to the drive app.
- On web, `HtmlIframeWidget` sets no explicit `sandbox` attribute. The drive app needs full scripting capability to run its UI. Origin-level isolation is enforced at the `postMessage` layer (ADR-0096 §6).
- `PointerInterceptor` is required on web; omitting it causes the iframe to be unresponsive to touch/click events.

## Consequences

- No new package dependency — `flutter_inappwebview` is already used by the rich-text composer editor; `HtmlIframeWidget` is from `package:core`.
- Platform split via conditional export keeps the mobile and web variants fully independent — no `kIsWeb` branching inside business logic.
- The shell's fixed `maxWidth: 800 / maxHeight: 677` constraint gives a consistent desktop appearance but may clip very tall picker UIs on small screens.
- `DriveIntentFakePage` allows full end-to-end testing of the picker flow without a live drive server or token exchange.
