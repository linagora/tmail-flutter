Context — The Problem

The bug: subfolders survived Empty Trash

What users expected

Empty Trash → deletes all emails inside Trash
AND all child mailboxes (subfolders) under Trash.

What actually happened

Only emails were deleted. Subfolders were silently left behind — there was no code path in the old flow to delete them.

Why the old GetX flow couldn't fix this cleanly

  • emptyTrashFolderAction used consumeState(stream) — a single-stream pipe into the GetX observable
  • Sequential async (delete emails → then delete subfolders) is impossible in this model
  • Business logic (capability check, null guards, strategy selection) was already crowding the controller
  • Untestable without full GetX binding setup

Sealed Classes — Concept

What is a sealed class in Dart?

A sealed class closes its subtype hierarchy to a single library file. Every direct subtype must be declared in the same file. The compiler knows the complete, finite set at compile time.

// Declare the sealed base
sealed class EmptyTrashState { const EmptyTrashState(); }

// All subtypes → same file ✓
class EmptyTrashIdle    extends EmptyTrashState { ... }
class EmptyTrashLoading extends EmptyTrashState { ... }
class EmptyTrashSuccess extends EmptyTrashState { ... }
class EmptyTrashFailure extends EmptyTrashState { ... }

// In another file → compile error ✗
class MyCoolState extends EmptyTrashState { ... }

The key property: exhaustive pattern matching

Because the compiler knows every subtype, a switch on a sealed class must cover all cases. A missing case is a compile error, not a runtime crash.

switch (state) {
  case EmptyTrashLoading():
    // show progress bar
  case EmptyTrashSuccess(:final subfoldersStatus):
    // show toast, refresh mailbox
  case EmptyTrashFailure(:final exception):
    // show error toast
  case EmptyTrashIdle():
    break;
  // No default: needed — compiler guarantees coverage
}
Sealed Classes — Why

Why not abstract class or enum?

Approach Custom fields per variant Exhaustiveness enforced by compiler Verdict
abstract class + is checks Yes No — adding a subtype silently breaks unhandled chains at runtime Risky
enum No — all variants share the same type, no custom fields Yes Too rigid
sealed class Yes — each subtype is its own class Yes — missing case = compile error Best of both

Why not return bool from sub-operations?

// Tempting but dangerous — exception is silently discarded
Future<bool> _clearMailbox(...) async {
  try { ...; return true; }
  catch (e) { return false; }  // ← e is gone forever
}

A bool discards the exception. EmptyTrashFailure can't carry it to UrgentExceptionHandler. Sealed result types preserve exceptions through the pipeline without exposing them in the public API.

Sealed Classes — Two Hierarchies in This Feature

Public state vs private sub-operation results

Public — EmptyTrashState

Consumed by MailboxProviderListenerWidget

sealed class EmptyTrashState {}

class EmptyTrashIdle    extends EmptyTrashState {}
class EmptyTrashLoading extends EmptyTrashState {}

class EmptyTrashSuccess extends EmptyTrashState {
  final List<EmailId> clearedEmailIds;
  final MailboxId? mailboxId;
  final SubfoldersDeleteStatus subfoldersStatus;
}

class EmptyTrashFailure extends EmptyTrashState {
  final Object? exception;
}

Private — file-only result types

Used only inside execute(), never exported

sealed class _ClearEmailResult {}
class _ClearEmailSuccess extends _ClearEmailResult {
  final List<EmailId> emailIds;
}
class _ClearEmailFailure extends _ClearEmailResult {
  final Object? exception; // preserved ✓
}

sealed class _SubfoldersResult {}
class _SubfoldersAllDeleted   extends _SubfoldersResult {}
class _SubfoldersSomeDeleted  extends _SubfoldersResult {}
class _SubfoldersDeleteFailed extends _SubfoldersResult {
  final Object? exception;
}

// Exhaustive mapping → compile-safe
subfoldersStatus = switch (subfoldersResult) {
  _SubfoldersAllDeleted()   => SubfoldersDeleteStatus.allDeleted,
  _SubfoldersSomeDeleted()  => SubfoldersDeleteStatus.someDeleted,
  _SubfoldersDeleteFailed() => SubfoldersDeleteStatus.failed,
};
Key Design Decision #1

Stream bridge: GetX → Riverpod

The bridge

// MailboxDashBoardController — before
emptyTrashFolderAction() {
  if (useJmapClear) {
    clearMailbox(session, accountId, ...);
  } else {
    consumeState(_interactor.execute(...));
  }
}

// After — controller just emits
emptyTrashFolderAction() {
  _emptyTrashStreamCtrl.add(trashFolder); // ← one line
}

// Riverpod wraps the stream
@riverpod
Stream<PresentationMailbox> emptyTrashRequested(Ref ref) =>
    Get.find<MailboxDashBoardController>()
        .onEmptyTrashRequested;

Why a broadcast stream, not a direct call?

Three code paths trigger empty-trash: mailbox panel, thread view banner, search mailbox controller. All already funnel through emptyTrashFolderAction. A stream means zero callers change — the Riverpod layer reacts automatically.

The Notifier — pure async orchestrator

@riverpod
class EmptyTrashNotifier extends _$EmptyTrashNotifier {

  @override
  EmptyTrashState build() => const EmptyTrashIdle();

  Future<void> execute(session, accountId,
      trashMailbox, childIds, useJmapClear) async {
    if (_isExecuting) return;  // guard double-tap
    state = const EmptyTrashLoading();

    final emailResult = trashMailbox.countTotalEmails == 0
        ? _ClearEmailSuccess(emailIds: const [])
        : useJmapClear
            ? await _clearMailbox(...)
            : await _emptyTrashByEmailDeletion(...);

    if (!mounted) return;  // guard after await

    if (childIds.isNotEmpty) {
      final subfoldersResult = await _deleteSubfolders(...);
      if (!mounted) return;
      // map result → subfoldersStatus
    }

    state = EmptyTrashSuccess(...);
  }
}
Key Design Decision #2 & #3

mounted guard & skip-empty optimisation

ref.mounted guard after every await

final emailResult = await _clearMailbox(...);

if (!mounted) return;   // ← user may have navigated away

if (emailResult is _ClearEmailFailure) {
  state = EmptyTrashFailure(...);
  return;
}

final subfoldersResult = await _deleteSubfolders(...);

if (!mounted) return;   // ← guard again before final write

state = EmptyTrashSuccess(...);

Without this guard: if the user navigates away mid-operation, writing to state on a disposed notifier throws. ref.mounted is Riverpod's built-in answer to the problem StatefulWidget.mounted solves in Flutter.

Skip email clearing when Trash is empty

final emailResult =
    trashMailbox.countTotalEmails == 0
    // Skip the network call entirely
    ? _ClearEmailSuccess(emailIds: const [])
    : useJmapClear
        ? await _clearMailbox(...)
        : await _emptyTrashByEmailDeletion(...);

Bug fixed by this

Previously: Trash with 0 emails but with subfolders → exited early with a warning toast, subfolders never deleted.

Now: 0 emails → skip email step → proceed to subfolder deletion normally.

Key Design Decision #4

MailboxProviderListenerWidget — the two-way bridge

class MailboxProviderListenerWidget
    extends ConsumerWidget {

  @override
  Widget build(BuildContext context, WidgetRef ref) {

    // ① Listen to trigger → call notifier
    ref.listen(emptyTrashRequestedProvider, (_, val) {
      val.whenData((trash) => _executeEmptyTrash(...));
    });

    // ② Listen to state → bridge to GetX
    ref.listen(emptyTrashProvider, (_, next) {
      _handleEmptyTrashStateChange(context, next);
    });

    return child; // never rebuilds child
  }
}

ref.listen vs ref.watch

ref.listen reacts to state changes via a callback — it does not trigger a widget rebuild. The mailbox list beneath it remains untouched.

State → GetX side effects

switch (state) {
  case EmptyTrashLoading():
    dashboardCtrl.viewStateMailboxActionProgress
        .value = Right(ClearingMailbox());

  case EmptyTrashSuccess(:final clearedEmailIds,
                          :final subfoldersStatus):
    dashboardCtrl.viewStateMailboxActionProgress
        .value = Right(UIState.idle);
    dashboardCtrl.handleDeleteEmailsInMailbox(...);
    _showSuccessToast(context, subfoldersStatus);
    dashboardCtrl.dispatchMailboxUIAction(
        RefreshAllMailboxAction());

  case EmptyTrashFailure(:final exception):
    _showErrorToast(context);
    getBinding<UrgentExceptionHandler>()
        ?.handleUrgentException(exception: exception);

  case EmptyTrashIdle():
    break;
}
Architecture — Component Map

Who owns what

Component Type Responsibility
MailboxDashBoardController.
_emptyTrashStreamController		 GetX Trigger source — emits PresentationMailbox when the user confirms
emptyTrashRequestedProvider Riverpod Wraps the stream so ConsumerWidgets can ref.listen to it
EmptyTrashNotifier Riverpod Pure async orchestrator — sequences email clearing → subfolder deletion, owns EmptyTrashState
MailboxProviderListenerWidget ConsumerWidget Two-way bridge — listens to Riverpod, calls back into GetX side effects (toasts, progress, refresh)
MailboxItemConsumerWidget Flutter Extracts Obx(MailboxItemWidget(...)) to resolve widget hierarchy conflict when ConsumerWidget wraps the list
UrgentExceptionHandler Abstract Contract for fatal error escalation — keeps MailboxProviderListenerWidget decoupled from the concrete GetX implementation

Layer rule (from ADR-0092): empty_trash_provider.dart lives in the presentation layer, not domain or data. Providers use @riverpod (auto-dispose) — kept alive by the persistent ref.listen in MailboxProviderListenerWidget.

Unit Tests

Testing EmptyTrashNotifier without GetX

// No GetX bindings needed
setUp(() {
  container = ProviderContainer(overrides: [
    // Inject mocks via GetX.put() only
    // for the interactors the notifier reads
  ]);
});

test('success with JMAP clear and no subfolders', () async {
  final notifier =
      container.read(emptyTrashProvider.notifier);

  await notifier.execute(
    session, accountId, trashMailbox,
    [], /* childIds */
    true, /* useJmapClear */
  );

  expect(container.read(emptyTrashProvider),
    isA<EmptyTrashSuccess>().having(
      (s) => s.subfoldersStatus,
      'subfoldersStatus',
      SubfoldersDeleteStatus.none,
    ));
});

Covered scenarios

ScenarioExpected state
Emails + JMAP clear, no subfolders Success / none
Emails + email-deletion path, no subfolders Success / none
No emails, has subfolders Success / allDeleted
Email clearing fails Failure
Email clear OK, subfolders partial fail Success / someDeleted
Already executing (double-tap guard) Stays Loading, no re-execute

Zero GetX overhead in tests

Each test creates a ProviderContainer, overrides interactors with mocks, and reads the notifier directly. No routing, no binding registration, no controller lifecycle.

Consequences

Trade-offs & migration notes

Positive

Subfolder deletion is sequenced deterministically after email clearing; all outcomes surfaced to the user via differentiated toasts

MailboxDashBoardController loses one interactor dependency and its entire email-clearing branch — the controller is thinner

EmptyTrashNotifier is independently unit-testable with no GetX binding setup

mounted guard + UrgentExceptionHandler contract established as conventions for future Riverpod notifiers

Constraints / future work

emptyTrashRequestedProvider still calls Get.find<MailboxDashBoardController>() — intentional for the transition; will be removed when the controller migrates to Riverpod

MailboxProviderListenerWidget writes back to viewStateMailboxActionProgress (a GetX Rx) — deliberate two-way bridge, disappears when progress observation migrates

MailboxItemConsumerWidget is a wrapper whose only job is resolving a hierarchy conflict — unnecessary once BaseMailboxView is a ConsumerWidget

Summary

Three patterns, one feature

① Stream Bridge

GetX controller emits a PresentationMailbox event. Riverpod wraps the stream as a provider. Multiple callers → zero changes. The boundary is clean and explicit.

② @riverpod Notifier

Owns the sequential async logic. mounted guard after every await. No GetX dependency. Pure, testable orchestration with an explicit state machine.

③ Sealed State

Public EmptyTrashState + private _ClearEmailResult / _SubfoldersResult. Every switch is compiler-enforced. Exceptions travel through the pipeline without being lost.
